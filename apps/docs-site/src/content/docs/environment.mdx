<p className="doc-eyebrow">Configuration</p>

<p className="doc-lede">First-party apps call <code>{"bootstrapAskDbEnv()"}</code> from <code>{"@askdb/config"}</code>, which loads a local <code>{".env"}</code> (via dotenv), discovers <code>{"askdb.config.*"}</code> or <code>{".config/askdb.*"}</code>, and evaluates a nested <code>{"defineConfig({ ... } satisfies AskDbConfig)"}</code> export. That produces an <strong>in-memory runtime snapshot</strong> (structured config + flattened canonical map + <code>{"aiEnv"}</code> for <code>{"@askdb/core"}</code>). AskDB does <strong>not</strong> copy the full flattened map into <code>{"process.env"}</code>; use <code>{"getAskDbRuntimeConfig()"}</code> instead.</p>

## When to use this package

<ul>
  <li>You run first-party binaries (CLI, HTTP API, Studio) or your own Node host and want Prisma-style config discovery plus a typed runtime snapshot.</li>
  <li>You need <code>{"env(\"VAR\")"}</code> helpers and <code>{"defineConfig"}</code> nesting without mutating global <code>{"process.env"}</code> for every AskDB key.</li>
</ul>

<p>Spec and rationale: <a href="https://github.com/Ygilany/AskDB/blob/main/docs/adrs/0005-askdb-config-and-env-bootstrap.md">ADR 0005 — askdb config and env bootstrap</a>.</p>

## Bootstrap order

<ol>
  <li><code>{"dotenv"}</code> loads <code>{".env"}</code> (missing file is normal).</li>
  <li>
    If present, <code>{"askdb.config.*"}</code> or <code>{".config/askdb.*"}</code> is evaluated. The default export must be <code>{"defineConfig(...)"}</code> (legacy flat maps are rejected). Values are flattened to canonical keys and stored with the structured object.
  </li>
  <li>
    A small allowlist of non-empty shell variables (for example <code>{"PORT"}</code>, <code>{"HOST"}</code>, <code>{"ASKDB_MOCK_SQL"}</code>, <code>{"ASKDB_CORRELATION_ID"}</code>, <code>{"ASKDB_PRISMA_SCHEMA"}</code>, <code>{"ASKDB_INTROSPECT_OUT"}</code>, <code>{"DATABASE_URL"}</code>) may override matching flat entries at bootstrap time so CI and subprocesses can inject overrides without editing the config file.
  </li>
</ol>

<p>This mirrors the ergonomics of <a href="https://www.prisma.io/docs/orm/reference/prisma-config-reference">Prisma’s config reference</a>, but AskDB uses <code>{".config/askdb.*"}</code> so it does not collide with Prisma’s own <code>{".config/prisma.*"}</code> files.</p>

## `askdb init`

```
pnpm exec askdb init
# writes ./askdb.config.ts only (no .env file)
```

<p><code>{"askdb init"}</code> emits a grouped <code>{"defineConfig({ ... } satisfies AskDbConfig)"}</code> in <code>{"askdb.config.ts"}</code>. Example <code>{".env"}</code> keys and setup notes from the old split template are preserved as <strong>comments</strong> in that file. Use <code>{"env(\"VAR\")"}</code> for values you keep in <code>{".env"}</code> (loaded via <code>{"import \"dotenv/config\""}</code>) or in the shell environment; when a variable is unset, <code>{"flattenAskDbConfig"}</code> applies documented defaults (chat model, introspection output dir, database URL fallbacks, RAG embedding dimensions, and similar — see <code>{"@askdb/config"}</code>).</p>

```
import "dotenv/config";
import { defineConfig, env, type AskDbConfig } from "@askdb/config";

export default defineConfig({
  ai: {
    provider: "openai",
    providerConfig: {
      openai: {
        apiKey: "",
        model: env("MY_CHAT_MODEL"),
      },
    },
  },
  database: {
    provider: "postgres",
    providerConfig: {
      postgres: { databaseUrl: env("MY_DATABASE_URL") },
    },
  },
  introspection: {
    provider: "postgres",
    providerConfig: {
      postgres: {},
    },
    outputDir: env("MY_INTROSPECT_OUTPUT_DIR"),
  },
  rag: {
    embedder: "mock",
    embedderConfig: {},
    store: "memory",
    storeConfig: { memory: {} },
  },
} satisfies AskDbConfig);
```

<p>Omit <code>{"introspection.providerConfig.postgres.databaseUrl"}</code> to reuse <code>{"database.providerConfig.postgres.databaseUrl"}</code> as <code>{"DATABASE_URL"}</code> for live Postgres introspection.</p>

<p><code>{"introspection.outputDir"}</code> maps to <code>{"ASKDB_INTROSPECT_OUT"}</code> in the flattened snapshot. After bootstrap, <code>{"askdb introspect"}</code> reads the runtime snapshot (including <code>{"ASKDB_INTROSPECT_OUT"}</code> and <code>{"ASKDB_PRISMA_SCHEMA"}</code> via <code>{"getAskDbRuntimeConfig().ai.aiEnv"}</code>) to default <code>{"--out"}</code> or <code>{"--prisma-schema"}</code> when those flags are omitted.</p>

## Config discovery

<p>From the working directory (usually the process cwd):</p>

<ul>
  <li><code>{"askdb.config.<ext>"}</code> — extension precedence: <code>{"ts"}</code> → <code>{"mts"}</code> → <code>{"cts"}</code> → <code>{"js"}</code> → <code>{"mjs"}</code> → <code>{"cjs"}</code> (first existing file wins).</li>
  <li><code>{".config/askdb.<ext>"}</code> — same extension precedence if no root config exists.</li>
</ul>

## Common canonical variables

<p>These names are what <code>{"flattenAskDbConfig"}</code> emits and what appears on <code>{"getAskDbRuntimeConfig().flat"}</code> / <code>{"getAskDbRuntimeConfig().ai.aiEnv"}</code>. You normally set underlying secrets in <code>{".env"}</code> and wire them through <code>{"env(\"MY_KEY\")"}</code> inside <code>{"askdb.config.ts"}</code>, or rely on defaults where documented.</p>

<div className="definition-list">
            <div><code>{"OPENAI_API_KEY"}</code><span>Required for live NL-to-SQL generation (unless using mocks).</span></div>
            <div><code>{"OPENAI_BASE_URL"}</code><span>Optional custom OpenAI-compatible base URL.</span></div>
            <div><code>{"ASKDB_MODEL"}</code><span>Optional model ID; defaults are package-specific.</span></div>
            <div><code>{"DATABASE_URL"}</code><span>Required only for live Postgres introspection.</span></div>
            <div><code>{"ASKDB_INTROSPECT_OUT"}</code><span>Default output directory for <code>{"askdb introspect"}</code> when <code>{"--out"}</code> is omitted (set from nested <code>{"introspection.outputDir"}</code>).</span></div>
            <div><code>{"ASKDB_PRISMA_SCHEMA"}</code><span>Default Prisma schema path for <code>{"askdb introspect --engine prisma"}</code> when <code>{"--prisma-schema"}</code> is omitted.</span></div>
            <div><code>{"ASKDB_SCHEMA_PATH"}</code><span>Default schema path for CLI and HTTP server workflows.</span></div>
            <div><code>{"ASKDB_LOG_LEVEL"}</code><span>Structured log level.</span></div>
            <div><code>{"ASKDB_MODE"}</code><span>Default operating mode.</span></div>
            <div><code>{"ASKDB_OMIT_SENSITIVE_FROM_PROMPT"}</code><span>Omit sensitive identifiers from generation prompts when set to a truthy value.</span></div>
          </div>

## TUI Suggestions

<p><code>{"@askdb/tui"}</code> resolves AI settings from the same runtime snapshot after <code>{"bootstrapAskDbEnv()"}</code>. When an OpenAI-compatible key is configured (typically via <code>{"askdb.config"}</code> / <code>{".env"}</code>), pressing <code>{"g"}</code> requests AI enrichment suggestions. Without a key, the TUI still supports manual authoring and saving.</p>

## RAG Embeddings

<p><code>{"askdb-rag"}</code> defaults to a deterministic mock embedder for smoke tests. Set <code>{"OPENAI_API_KEY"}</code> and pass <code>{"--embedder openai"}</code> when you want real embeddings; pass <code>{"--pg-url"}</code> only when using the pgvector store.</p>
