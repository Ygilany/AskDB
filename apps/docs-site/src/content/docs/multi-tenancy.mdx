<p className="doc-eyebrow">Tenant-scoped SQL generation</p>

<p className="doc-lede">AskDB's multi-tenancy proof lets you define a tenant policy that scopes NL→SQL generation to a tenant boundary — ensuring generated SQL always includes tenant-filtering predicates.</p>

## What multi-tenancy means in AskDB

AskDB does not execute SQL. What it does is inject tenant-scoping predicates into generated SQL so that every query you receive back is already constrained to the authorized tenant boundary. Your application owns execution, audit logging, and row-level enforcement.

The tenant policy lives in `tenant-policy.md` at the root of the schema artifact directory, alongside `schema.json` and `concepts.md`. It defines how your schema represents tenancy so AskDB can carry that model into prompt assembly and SQL validation.

## Tenant policy concepts

<div className="definition-list">
  <div><strong>Tenant roots</strong><span>The primary tenant-identifying table (e.g. <code>{"organizations"}</code>) and its <code>{"tenantIdColumn"}</code>. A schema may have more than one root when tenancy is layered (agency → sub-agency → account).</span></div>
  <div><strong>Hierarchy</strong><span>Parent→child foreign-key relationships between tenant roots, expressing nested access structures such as agency → sub-agency → client.</span></div>
  <div><strong>Scoped tables</strong><span>Tables that belong to a tenant via a foreign key to a tenant root or to another scoped table. Generated SQL always includes a join or predicate that constrains rows to the authorized tenant scope.</span></div>
  <div><strong>Polymorphic tables</strong><span>Tables that reference multiple entity types using a type discriminator column (e.g. <code>{"owner_type"}</code> / <code>{"owner_id"}</code>). AskDB handles the discriminator when generating tenant predicates.</span></div>
  <div><strong>Global tables</strong><span>Tenant-agnostic lookup or reference tables (e.g. <code>{"currencies"}</code>, <code>{"countries"}</code>). These are never tenant-filtered.</span></div>
</div>

## SQL output modes

<div className="definition-list">
  <div><code>{"sql-only"}</code><span>Tenant predicates are inlined as literals. Useful for development inspection and ad-hoc queries where you want to read the full SQL directly.</span></div>
  <div><code>{"sql-params"}</code><span>Tenant values are extracted as positional <code>{"$N"}</code> parameters. Suitable for parameterized query execution in application code.</span></div>
</div>

## Authoring the policy

The recommended path is Studio's **Tenancy** view, which lets you define tenant roots, hierarchy links, scoped tables, polymorphic tables, and global tables through a browser UI. Studio also offers AI-assisted policy drafting: describe your schema's multi-tenancy structure in plain language and AskDB will propose a `tenant-policy.md` for you to review and confirm.

You can also edit `tenant-policy.md` directly. The file uses YAML frontmatter for the structured policy and an optional free-text body for notes.

Example `tenant-policy.md` frontmatter for a simple SaaS schema:

```yaml
---
tenantRoots:
  - table: organizations
    tenantIdColumn: id

scopedTables:
  - table: projects
    tenantForeignKey: organization_id
    tenantRoot: organizations
  - table: documents
    tenantForeignKey: organization_id
    tenantRoot: organizations

globalTables:
  - table: plan_tiers
  - table: countries
---
```

With this policy in place, a question like "how many documents were created this month?" generates SQL that always includes a `WHERE organization_id = $1` (or an inlined literal in `sql-only` mode) rather than scanning the full table.

## Testing in Studio

The **Ask** panel in Studio includes tenant scope controls. Provide a JSON tenant scope — specifying the access kind, the tenant root, and the tenant IDs — before generating SQL to see how the predicate injection works:

```json
{
  "kind": "exact",
  "tenantRoot": "organizations",
  "tenantIds": ["org_abc123"]
}
```

Switch between `sql-only` and `sql-params` output modes in the same panel to compare the two forms of the generated SQL.

## Schema artifact layout

A schema artifact directory with a tenant policy looks like this:

```
my-app.schema/
  schema.json          # physical schema (tables, columns, foreign keys)
  concepts.md          # optional domain glossary
  tenant-policy.md     # tenant roots, hierarchy, scoped/polymorphic/global tables
  tables/
    organizations.md
    projects.md
    documents.md
```

`tenant-policy.md` is optional. When absent, AskDB generates SQL without any tenant-scoping predicates — the same behavior as before Phase 10.
