<p class="doc-eyebrow">Reference ┬À Configuration</p>

<p class="doc-lede">AskDB resolves config from two sources: an <code>askdb.config.*</code> file (TypeScript or JavaScript) and environment variables. Both are loaded by <code>@askdb/config</code> at startup into an in-memory runtime snapshot.</p>

## `askdb.config.ts`

`askdb init` scaffolds this file. It uses `defineConfig` for type safety:

```ts
import { defineConfig, env } from "@askdb/config";

export default defineConfig({
  ai: {
    provider: "openai",
    providerConfig: {
      openai: {
        apiKey: env("OPENAI_API_KEY"),
        model: env("OPENAI_MODEL"),
      },
    },
  },
  introspection: {
    provider: "postgres",
    providerConfig: {
      postgres: { databaseUrl: env("DATABASE_URL") },
    },
    outputDir: "./my-app.schema",
  },
  rag: {
    embedder: "openai",
    store: "pgvector",
    storeConfig: {
      pgvector: { databaseUrl: env("ASKDB_PGVECTOR_URL") },
    },
  },
  logging: {
    level: "info",
  },
});
```

Use the `env()` helper (rather than `process.env`) so AskDB can capture which environment variables your config depends on and apply its defaults pass.

`defineConfig` returns the object unchanged at runtime ÔÇö it exists only for type inference. The full type is `AskDbConfig`.

## File discovery

`bootstrapAskDbEnv()` looks for config in this order:

1. `askdb.config.ts`
2. `askdb.config.mts`
3. `askdb.config.cts`
4. `askdb.config.js`
5. `askdb.config.mjs`
6. `askdb.config.cjs`
7. `askdb.config.json`
8. `.config/askdb.*` (any of the above extensions)

The first match wins. Looks in the current working directory, then walks up to find a project root.

## Environment variables

AskDB reads `.env` next to your process via `dotenv`, then layers explicit env vars on top. The most common keys:

Variable names you control. The variables listed below are the ones the scaffolded `askdb.config.ts` reads via `env("...")`; rename them in your config and `.env` together if you'd rather use your own scheme.

| Variable | Maps to (config path) | Notes |
| --- | --- | --- |
| `OPENAI_API_KEY` | `ai.providerConfig.openai.apiKey` | Standard OpenAI key. |
| `OPENAI_MODEL` | `ai.providerConfig.openai.model` | Optional model override. Default: `gpt-4o-mini`. |
| `ANTHROPIC_API_KEY` | `ai.providerConfig.anthropic.apiKey` | When `ai.provider = "anthropic"`. Default model: `claude-sonnet-4-6`. |
| `AZURE_API_KEY`, `AZURE_RESOURCE_NAME` | `ai.providerConfig.azure.*` | Azure OpenAI. |
| `ASKDB_INTROSPECT_POSTGRES_URL` | `introspection.providerConfig.postgres.databaseUrl` | Postgres introspection URL. Still falls back to `DATABASE_URL` if unset. |
| `ASKDB_INTROSPECT_MYSQL_URL` | `introspection.providerConfig.mysql.databaseUrl` | MySQL introspection URL. No `DATABASE_URL` fallback. |
| `ASKDB_INTROSPECT_SQLSERVER_URL` | `introspection.providerConfig.sqlserver.databaseUrl` | SQL Server introspection URL. No `DATABASE_URL` fallback. |
| `ASKDB_INTROSPECT_SQLITE_FILE` | `introspection.providerConfig.sqlite.file` | SQLite file path. |
| `ASKDB_INTROSPECT_OUT` | `introspection.outputDir` | Default schema-artifact output directory. |
| `ASKDB_STUDIO_DATABASE_URL` | `studio.execute.databaseUrl` | Connection for the Studio playground query runner. |
| `ASKDB_SCHEMA_PATH` | `host.schemaPath` | Schema artifact the HTTP server loads at startup. |
| `ASKDB_PGVECTOR_URL` | `rag.storeConfig.pgvector.databaseUrl` | Connection string for the pgvector store. |
| `ASKDB_OMIT_SENSITIVE_FROM_PROMPT` | `modes.omitSensitiveFromPrompt` | When `true`, omit sensitive columns from the prompt. |
| `ASKDB_LOG_LEVEL` | `logging.level` | One of `trace`, `debug`, `info`, `warn`, `error`, `fatal`, `silent`. |
| `ASKDB_MODE` | `modes.askdbMode` | Default operating mode (`schema_only` / `bounded_results`). |

The complete list of env keys is generated from the config schema and exposed as `ASKDB_ENV_PROJECTION` in `@askdb/config`.

## Custom providers

`ai.provider` accepts any string ÔÇö not just the first-party literals. When an unknown provider string is used, `flattenAskDbConfig` applies the generic branch: `ai.providerConfig.custom.{apiKey,baseUrl,model}` flatten to the universal `ASKDB_AI_API_KEY` / `ASKDB_AI_BASE_URL` / `ASKDB_AI_MODEL` env keys.

```ts
import { defineConfig, env } from "@askdb/config";

export default defineConfig({
  ai: {
    provider: "mistral",           // any string
    providerConfig: {
      custom: {
        apiKey: env("MISTRAL_API_KEY"),
        model: "mistral-large-latest",
      },
    },
  },
  // ...
});
```

This works end to end only when the consuming registry has an adapter registered under that provider name. First-party apps (`@askdb/http-api`, the CLI) register `openai`, `azure`, `foundry`, `google`, and `anthropic`. For any other provider you have two options:

1. **Embed AskDB** and register your own adapter via `createAiRegistry` ÔÇö about 40 lines of code.
2. **Use the `openai` provider** with a custom `baseUrl` for any OpenAI-compatible endpoint (vLLM, OpenRouter, Together, Ollama, ÔÇª).

See [Bring your own model](/guides/bring-your-own-model/) for the embedding walkthrough.

## Reading config at runtime

In code that uses the AskDB libraries directly, call `bootstrapAskDbEnv()` at startup and then read settings via `getAskDbRuntimeConfig()`:

```ts
import { bootstrapAskDbEnv, getAskDbRuntimeConfig } from "@askdb/config";

bootstrapAskDbEnv({ cwd: process.cwd() });

const rt = getAskDbRuntimeConfig();
const aiEnv = rt.ai.aiEnv;
const schemaPath = rt.schema.path;
```

AskDB does **not** copy the full config into `process.env`. The runtime snapshot is the source of truth ÔÇö use `getAskDbRuntimeConfig` rather than `process.env` for anything AskDB-specific.

## Config precedence

For any given setting, the precedence (highest wins) is:

1. Function-call arguments (`ask({ mode: "..." })` etc.).
2. Environment variables.
3. `askdb.config.*` file.
4. Built-in defaults.

This lets you ship a config file with sensible defaults and override per-environment via env vars without rebuilding.
