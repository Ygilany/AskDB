## Core

### `@askdb/core`

The dialect-agnostic NL-to-SQL pipeline. Schema loading, prompt construction, validation, and tenant-scope handling all live here.

<InstallTabs pkgs="@askdb/core" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `ask(options)` | The main pipeline. Returns validated SQL. See [full API reference](/reference/core-api/). |
| `loadSchema(path)` | Load a schema artifact from a directory, bundle, or `schema.json`. |
| `validateTenantScope` / `resolveTenantSql` | Tenant-scope enforcement primitives. |
| `createAskDbLogger` | Structured logger that emits AskDB log events. |
| `POSTGRES_DIALECT` and built-in dialect IDs | Dialect inputs accepted by `ask()`. |
| `AskDbLanguageModel` | AskDB's public name for the AI SDK language model contract. |
| `AskUsage` | Token usage type returned in `AskPipelineResult`. |

See the **[@askdb/core API reference](/reference/core-api/)** for the full option and return-type documentation.

### `@askdb/client`

Config-aware facade over `@askdb/core`. `createAskDb()` resolves the schema, model, and
dialect from your `askdb.config.*` and the provider adapters you pass, so call sites pass
only a question. Install it when you want the shortest config-driven path; skip it when you
construct the model and load the schema yourself and call `ask()` directly.

<InstallTabs pkgs="@askdb/client @askdb/ai-openai @askdb/config" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `createAskDb(options)` | Build a client bound to a config + provider adapters (`providers: [openaiProvider]`; or a prebuilt `registry`). Returns `{ ask, reload }`. Options include `unknownDialect: "throw" \| "fallback-postgres"` for artifacts recorded by an unrecognized provider. |
| `client.ask(question, overrides?)` | Resolve schema/model/dialect and run the pipeline. Overrides: `schema`, `model`, `dialect`, plus all `ask()` options. |
| `AskDbClient` / `CreateAskDbOptions` / `AskOverrides` | Types for the client, constructor options, and per-call overrides. |
| `SchemaLoadError` / `SchemaNotConfiguredError` / `ModelNotConfiguredError` / `DialectNotSupportedError` | Typed errors for the resolution steps — catch these to map failures to your own error handling. |

See the **[@askdb/client API reference](/reference/client-api/)** for the full option documentation.

### `@askdb/ai`

AI provider registry and shared config helpers. You rarely install or import this directly:
`@askdb/client` builds the registry for you from `providers`, and the adapter packages pull
`@askdb/ai` in as a peer dependency. Import it yourself only to build a model object outside
the client (`createAiRegistry(...).createLanguageModelFromEnv(...)`) or to share one registry
across several clients.

<InstallTabs pkgs="@askdb/ai" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `createAiRegistry` | Register provider adapters and build language or embedding models. |
| `registry.resolveAiConfig(env)` / `registry.resolveEmbeddingConfig(env)` | Resolve provider config from env without constructing a model. These are methods on the registry returned by `createAiRegistry`, not standalone exports. |
| `resolveBaseConfig` | Low-level helper — resolves a provider's base config object from an env map. Used internally by adapters. |
| `withEmbeddingProviderOptions` | Wraps embedding provider options; shared across adapters. |
| `aiKeyMissingMessage` | Shared human-readable missing-key guidance. |

### Provider adapters

Provider adapters own concrete AI SDK provider dependencies. Each adapter declares `ai` and `@askdb/ai` as peer dependencies — npm/pnpm install them automatically, but your app should depend on `ai` directly when you want to pin the version.

Any provider that has a Vercel AI SDK package can be added as an adapter. Unknown `ai.provider` values require a registered adapter — first-party apps register `openai`, `azure`, `foundry`, `google`, and `anthropic`; for anything else, embed AskDB and register your own adapter, or use an OpenAI-compatible endpoint via the `openai` provider and a custom base URL.

### `@askdb/connectors`

Connector provider registry — mirrors the `@askdb/ai` pattern for introspection. Each engine adapter exports a `ConnectorProviderAdapter` constant; `@askdb/connectors` wires them into a unified registry used by the CLI and apps. Install directly only when you are building a custom introspection flow.

<InstallTabs pkgs="@askdb/connectors" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `createConnectorRegistry(adapters)` | Build a registry from a set of provider adapters. |
| `ConnectorRegistry` | Registry type — `createConnector(provider, config)`, `getTemplates(provider)`. |
| `ConnectorProviderAdapter` | The adapter shape each engine package exports. |

### `@askdb/config`

Optional config bootstrap: `.env` and `askdb.config.*` discovery, type-safe `defineConfig`, and a runtime snapshot accessor.

<InstallTabs pkgs="@askdb/config" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `defineConfig(config)` | Type-safe config builder. |
| `bootstrapAskDbEnv(options)` | Load `.env` + `askdb.config.*` and install the runtime snapshot. |
| `getAskDbRuntimeConfig()` | Read the runtime snapshot (preferred over `process.env`). |
| `discoverAskDbConfigPath()` | Programmatic config-file discovery. |
| `env`, `requiredEnv` | Typed env-var accessors. |

## Engine adapters

Each engine package ships the introspection connector for its engine, any catalog SQL templates, and a re-export of that engine's dialect constant. The dialect specs themselves (SQL generation + validation rules) live in `@askdb/core` — `ask()` needs only the dialect id, no engine package.

### `@askdb/postgres`

<InstallTabs pkgs="@askdb/postgres" />

PostgreSQL dialect, connector, catalog SQL templates, and an optional `pg`-based catalog query runner for live introspection. Pass `dialect: "postgres"` to `ask()`.

### `@askdb/mysql`

<InstallTabs pkgs="@askdb/mysql" />

MySQL dialect, connector, catalog SQL templates, and an optional `mysql2`-based catalog query runner for live introspection. Pass `dialect: "mysql"` to `ask()`.

### `@askdb/sqlite`

<InstallTabs pkgs="@askdb/sqlite" />

SQLite dialect, connector, catalog SQL templates, and an optional `better-sqlite3`-based catalog query runner for live introspection. Works with embedded file-backed databases. Pass `dialect: "sqlite"` to `ask()`.

### `@askdb/sqlserver`

<InstallTabs pkgs="@askdb/sqlserver" />

SQL Server (T-SQL) dialect, connector, catalog SQL templates, and an optional `mssql`-based catalog query runner for live introspection. Pass `dialect: "sqlserver"` to `ask()`.

## Introspection

### `@askdb/introspect`

The engine-agnostic introspection contract and the schema-directory writer. Use directly only when you're calling introspection from your own code — the CLI wraps this for you.

<InstallTabs pkgs="@askdb/introspect" />

### `@askdb/prisma`

File-only connector that reads `.prisma` schema files and produces AskDB physical metadata — no live database needed. Pair with an engine adapter for SQL generation at runtime.

<InstallTabs pkgs="@askdb/prisma @askdb/introspect" />

See [the Prisma integration guide](/guides/integrations/prisma/).

## Authoring

### `@askdb/studio`

Browser-based schema-authoring UI. The CLI's `askdb studio` command lazy-imports this package. Schema browsing, enrichment, and SQL generation do not require a database driver. The Query Playground's live execute feature supports Postgres, MySQL, SQLite, and SQL Server — each requires its own optional peer driver package.

<InstallTabs pkgs="@askdb/studio" />

| Execute provider | Package |
|---|---|
| Postgres | `pg` (optional peer) |
| MySQL | `mysql2` (optional peer) |
| SQLite | `better-sqlite3` (optional peer) |
| SQL Server | `mssql` (optional peer) |

Install only the package for your dialect. The execute provider is inferred from your active introspection provider unless overridden with `studio.execute.provider`.

`@askdb/studio` lists all four packages as dev dependencies for local package development and CI type-checking. This is not a requirement for your application — only install the driver you need at runtime.

See [Studio — driver packages](/studio/#driver-packages) and [Configuration reference — Studio execute](/reference/config/#studio-execute-configuration) for setup details.

### `@askdb/tui`

Terminal-based schema-authoring UI. The CLI's `askdb enrich` command lazy-imports this package.

<InstallTabs pkgs="@askdb/tui" />

### `@askdb/enrich`

Headless schema-authoring helpers shared by Studio and TUI. Use directly when building a custom authoring surface.

<InstallTabs pkgs="@askdb/enrich" />

## Retrieval

### `@askdb/rag`

Chunks the schema artifact, indexes it in a vector store, and retrieves relevant chunks per question. Use with `ask()`'s `retriever` option for large schemas.

<InstallTabs pkgs="@askdb/rag" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `buildSchemaIndex({ schema, store, embedder, ... })` | Chunk + embed + upsert a schema artifact; returns `{ retriever, stats, chunks }`. |
| `createAiSdkEmbedder({ model })` / `createOpenAiEmbedder(...)` | Wrap an AI SDK embedding model (or raw OpenAI) as the `embedder`. |
| `createRetriever({ store, embedder })` | Build the retriever passed to `ask()`. |
| `createPgvectorStore` | The pgvector store factory. |
| `loadChunkerSourcesFromDir(path)` | Load a schema directory into chunker format (includes markdown). |

Ships a small CLI binary `askdb-rag` with three commands — `index`, `query`, and `setup-store` (pgvector provisioning). See [CLI reference — the askdb-rag binary](/reference/cli/#the-askdb-rag-binary) and [RAG for large schemas](/guides/rag-for-large-schemas/).

See the **[@askdb/rag API reference](/reference/rag-api/)** for the full option and type documentation.

## Service surfaces

### `@askdb/http-api`

Minimal HTTP server wrapping `ask()`. Two endpoints (`/health`, `POST /ask`), designed to sit behind your gateway.

<InstallTabs pkgs="@askdb/http-api" />

**Key exports**

| Symbol | Purpose |
| --- | --- |
| `createAskDbHttpServer(options)` | Build the server. |
| `AskHttpRequest` / `AskHttpSuccessResponse` / `AskHttpErrorResponse` | Request and response types. |

See [the HTTP API reference](/reference/http-api/) for endpoints and error codes.

### `askdb` (CLI)

The `askdb` binary. Ships as a normal npm package — install for local development, demos, and CI.

<InstallTabs pkgs="askdb" />

See [the CLI reference](/reference/cli/).

## Dependency direction

| Package | Depends on |
| --- | --- |
| `@askdb/postgres`, `@askdb/mysql`, `@askdb/sqlite`, `@askdb/sqlserver` | `@askdb/core`, `@askdb/connectors` |
| `@askdb/prisma` | `@askdb/introspect`, `@askdb/connectors` |
| `@askdb/connectors` | `@askdb/introspect` |
| `@askdb/ai` | AI SDK core package (`ai`) |
| `@askdb/ai-openai`, `@askdb/ai-azure`, `@askdb/ai-google`, `@askdb/ai-anthropic` | `@askdb/ai` + the matching AI SDK provider package |
| `@askdb/enrich`, `@askdb/rag` | `@askdb/core` |
| `@askdb/tui`, `@askdb/studio` | `@askdb/enrich` (+ `@askdb/rag` when used) |
| `@askdb/http-api` | `@askdb/core` (+ engine adapter) |
| `askdb` (CLI) | All of the above (lazy-imported per command) |

Reusable packages stay small and layered. First-party surfaces (CLI, Studio, TUI, HTTP API) compose them.
