<p className="doc-eyebrow">Scale-up journey</p>

<p className="doc-lede">
  AskDB's pgvector store needs a <code>{"vector"}</code> extension, one table, and a handful of indexes before it can hold embeddings. This guide covers all three provisioning paths so you can pick the one that fits your deployment.
</p>

## Prerequisites

- A Postgres database with the [`pgvector` extension](https://github.com/pgvector/pgvector) available (version 0.5.0+).
- `@askdb/rag` installed, plus the `pg` driver if you are passing a connection string:

```
pnpm add @askdb/rag pg
```

- `ASKDB_PGVECTOR_URL` set to your connection string, or `rag.storeConfig.pgvector.databaseUrl` in `askdb.config.ts`.

## Which path to choose

<div className="definition-list">
  <div><strong>Studio (local dev)</strong><span>Zero config — Studio calls <code>{"ensureSchema()"}</code> automatically on every RAG operation. Nothing to run manually.</span></div>
  <div><strong><code>{"askdb-rag setup-store"}</code> (CI / staging)</strong><span>One command that provisions the schema idempotently. Good for Dockerfiles, GitHub Actions steps, and staging bootstrap scripts.</span></div>
  <div><strong><code>{"setupSql()"}</code> (production migrations)</strong><span>Returns the DDL as a string. Paste it into a versioned migration file so production DDL is explicit, reviewed, and tracked.</span></div>
</div>

## Path 1 — Studio (local dev)

Configure pgvector in `askdb.config.ts`:

```ts
export default {
  rag: {
    embedder: "openai",
    store: "pgvector",
    storeConfig: {
      pgvector: {
        databaseUrl: process.env.ASKDB_PGVECTOR_URL,
      },
    },
  },
} satisfies AskDbConfig;
```

Then start Studio:

```
askdb studio --schema ./my-app.schema
```

The first time any RAG operation runs (status check, index, or query), Studio calls `ensureSchema()` and the table is created. No extra step needed.

## Path 2 — CLI (CI and staging)

Run `setup-store` as part of your environment bootstrap. All DDL uses `IF NOT EXISTS` guards so the command is safe to re-run on every deploy.

```bash
askdb-rag setup-store \
  --pg-url "$DATABASE_URL" \
  --dimensions 1536
```

Options:

<div className="definition-list">
  <div><code>{"--pg-url"}</code><span>Required. Postgres connection string.</span></div>
  <div><code>{"--pg-table"}</code><span>Table name. Defaults to <code>{"askdb_rag_chunks"}</code>.</span></div>
  <div><code>{"--dimensions"}</code><span>Embedding dimensions. Defaults to <code>{"1536"}</code>. Must match the embedder you use for indexing.</span></div>
</div>

Example GitHub Actions step:

```yaml
- name: Provision pgvector schema
  run: askdb-rag setup-store --pg-url "${{ secrets.PGVECTOR_URL }}" --dimensions 1536
```

## Path 3 — Explicit migration (production)

Generate the DDL and add it to your migration system:

```ts
import { createPgvectorStore } from "@askdb/rag/stores/pgvector";

const store = createPgvectorStore({
  connectionString: process.env.DATABASE_URL!,
  dimensions: 1536,
});

console.log(store.setupSql());
```

The output looks like:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE IF NOT EXISTS "askdb_rag_chunks" (
  id text PRIMARY KEY,
  type text NOT NULL,
  text text NOT NULL,
  schema_id text NOT NULL,
  refs jsonb NOT NULL DEFAULT '[]'::jsonb,
  sensitive boolean NOT NULL DEFAULT false,
  embedding vector(1536) NOT NULL
);
CREATE INDEX IF NOT EXISTS "askdb_rag_chunks_schema_id" ON "askdb_rag_chunks" (schema_id);
CREATE INDEX IF NOT EXISTS "askdb_rag_chunks_type" ON "askdb_rag_chunks" (type);
CREATE INDEX IF NOT EXISTS "askdb_rag_chunks_refs" ON "askdb_rag_chunks" USING gin (refs);
CREATE INDEX IF NOT EXISTS "askdb_rag_chunks_embedding_hnsw" ON "askdb_rag_chunks" USING hnsw (embedding vector_cosine_ops);
```

Copy this into a migration file in Flyway, Liquibase, Drizzle, Prisma, or any runner your team uses.

### Changing dimensions or index strategy

If you change `dimensions` or `indexStrategy` after the table already exists, `IF NOT EXISTS` means `CREATE TABLE` and `CREATE INDEX` will not re-run. Drop and recreate the table (or add a new migration) to apply structural changes.

## Using a custom table name

Pass `--pg-table` (CLI) or `table` (API) to isolate schemas from different AskDB deployments in the same database:

```bash
askdb-rag setup-store --pg-url "$DATABASE_URL" --pg-table myapp_rag_chunks --dimensions 1536
```

```ts
const store = createPgvectorStore({
  connectionString: process.env.DATABASE_URL!,
  table: "myapp_rag_chunks",
  dimensions: 1536,
});
await store.ensureSchema();
```

Match the same table name in `askdb.config.ts` under `rag.storeConfig.pgvector.table`.
